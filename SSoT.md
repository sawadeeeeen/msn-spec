# msn-spec (GDD AI協調スキルパッケージ)

## 1. 概要
- 本システムは、保証駆動開発（GDD）に基づいて、3つの自律的なAIエージェント（Messi, Suárez, Neymar）が連携し、高品質かつ整合性のあるシステム仕様書（SSoT）を構築・維持するためのAIスキルパッケージである。
- 本システムは、複数機能の並行開発およびコンテキストトークンの最適化のため、**「機能単位の複数ファイル化」**および**「ローカル隠しフォルダでのレポート管理」**を採用する。

## 2. システム境界 (System Boundary)

本パッケージは、対象プロジェクトのルートディレクトリにおいて、以下の命名規則に基づいてファイルを介し、情報の受け渡しを行う。

### 2.1. 命名規則と格納場所 (Naming & Location)
- **仕様書（SSoT）**: `[機能名]-SSoT.md`（例: `auth-SSoT.md`）
  - **格納ディレクトリ**: デフォルトは **`docs/specs/`** とする。
  - **自動探索フォールバック**: すでにプロジェクト内に他の階層で `*SSoT.md` が存在する場合、またはユーザーから明示的なフォルダ指定がある場合は、既存および指定されたフォルダを最優先で探索・使用する。
- **監査テストレポート**: `.agents/gdd-reports/[機能名]-gdd-validator-report.md` ※プロジェクトルートを汚さない隠しフォルダ配下。

### 2.2. 各コンポーネントの入出力
- **messi (Messi / 仕様作成)**:
  - インプット: ユーザー要求、既存のコード/ドキュメント、既存の `[機能名]-SSoT.md`
  - アウトプット: `[機能名]-SSoT.md`
- **suarez (Suárez / 品質監査)**:
  - インプット: `[機能名]-SSoT.md`
  - アウトプット: `.agents/gdd-reports/[機能名]-gdd-validator-report.md`
- **neymar (Neymar / 自動修復)**:
  - インプット: `[機能名]-SSoT.md`、`.agents/gdd-reports/[機能名]-gdd-validator-report.md`
  - アウトプット: 修復された `[機能名]-SSoT.md`

## 3. 表明（Guarantees）

### 3.1. 事前条件 (Pre-conditions)
- **パッケージインストール**: npx skills 規格およびカスタムスキルをサポートするAIエージェント環境の設定ディレクトリに、3つのスキル（`messi`, `suarez`, `neymar`）が配置されていること。
- **実行コンテキスト**: コマンドの実行は、対象プロジェクトのルートディレクトリであること（ファイル参照を正しく通すため）。

### 3.2. 事後条件 (Post-conditions)
- **ファイル更新の永続化**: 各スキルの実行完了時、作成または修正されたファイル（`[機能名]-SSoT.md`、`.agents/gdd-reports/[機能名]-gdd-validator-report.md`）は必ずローカルディスクに保存（永続化）され、別セッションに引き継げる状態になること。
- **レポートの非公開化**: 監査テストレポートは、`.agents/` などの隠しフォルダに保存されるため、Gitにコミットおよび共有されないこと（`.gitignore` で除外されていること）。

### 3.3. 不変条件 (Invariants / パッケージが常に維持すべきルール)
- **SSoTの純度 (What vs How)**: `[機能名]-SSoT.md` にには「何をするか（What）」のみを定義し、「どう実装するか（How）」を含めてはならない。
- **対話と進捗の可視化**: `messi` は、仕様作成時に質問を投げる際、必ず全体のステップ数と現在の進捗（例: `[1/3]`）を提示すること。
- **対話の最大上限 (3ターン制限)**: `messi` の対話フェーズにおける質問ラリーは**最大3回（3ターン）**までとし、それに達した場合はその時点の合意事項に基づき必ず `[機能名]-SSoT.md` を書き出すこと（未解決の細部は後流の `suarez` に委ねる）。
- **レポートの標準化と全項目監査**: `suarez` が出力するレポートは、必ず「総合判定（`PASS` / `FAIL`）」を明記し、`3.3. 不変条件` に定義されたすべての項目（What vs How, 3ターン制限等）に対する個別の検証結果（監査結果）を網羅的に記載すること。エラー時は `[ERROR-XX]` という一意の識別IDを付与すること。
- **完全修復の義務**: `neymar` は、対応する `[機能名]-gdd-validator-report.md` に記載された `[ERROR-XX]` のすべての指摘事項を論理的に解消し、漏れなく修復すること。

## 4. 正常系ユースケース (Happy Paths)

```
[Managerの要求] ──> ① messi (対話後、SSoT.md 作成)
                      └──> ② suarez (監査実行)
                            ├──> [判定: PASS] ──> 最終検収 (完了)
                            └──> [判定: FAIL] ──> ③ neymar (修復)
                                                   └──> ② suarez へ戻る
```

### ユースケース 1: 新規仕様作成と検証
1. ユーザー（Manager）が `messi` に新規機能名（例: `auth`）と要求を渡す。
2. `messi` が質問と進捗を表示し、ユーザーとの合意を形成する（最大3ターン）。
3. `messi` が `docs/specs/auth-SSoT.md`（または自動探索されたフォルダ）に仕様書を作成する。
4. ユーザーが `suarez` を実行し、`auth-SSoT.md` を監査させる。
5. `suarez` が全不変条件の監査を行い、`.agents/gdd-reports/auth-gdd-validator-report.md` を出力する（判定: `PASS`）。
6. ユーザーが仕様を確定（検収）する。

### ユースケース 2: 監査によるエラー検出と自動修復
1. `suarez` の監査で不備が検出され、`.agents/gdd-reports/auth-gdd-validator-report.md` が出力される（判定: `FAIL`）。
2. ユーザーが `neymar` に対し、`auth-SSoT.md` とエラーレポートを元にした修復を指示する。
3. `neymar` がすべてのエラーを解消し、修正した `auth-SSoT.md` を出力する。
4. ユーザーが再度 `suarez` を実行し、総合判定が `PASS` になったことを確認して検収する。

---

## 5. 異常系とエラー処理 (Exceptional Paths)

### 5.1. ユーザー入力が極端に曖昧な場合
- **対処**: `messi` は勝手に仕様を作らず、最も確認が必要な1〜2問の質問と推奨案を提示し、入力を促す。

### 5.2. 修復ループの無限実行（自動実行時の保護機構）
- **対処**: 自動実行および対話による更新ループは **最大3回** までとする。3回繰り返しても `FAIL` が解消しない場合、システムは処理を強制終了し、これまでの修復ログと未解決のエラーレポートを明示して、ユーザー（Manager）に制御を戻す。

### 5.3. ディスクへのファイル書き込み（永続化）に失敗した場合
- **対処**: メモリ上の変更状態を破棄（ロールバック）し、書き込みに失敗した旨の重大警告（エラーメッセージ）を表示して直ちにすべての処理を中断する。
