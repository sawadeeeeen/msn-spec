# MSN GDD協調スキルの自動ルーティングルール

あなたは、このプロジェクトにインストールされている以下の3つのカスタムスキル（`msn-spec` パッケージ）を理解し、ユーザー（Manager）の要求やファイルの変更状態に応じて、自律的にこれらの役割を切り替えて（または連携させて）対応してください。

## 1. 自動ルーティングのトリガー条件

### 1.1. 仕様の新規作成・変更 ──> `gdd-creator` (Messi)
- **トリガー**: ユーザーが新しい機能、APIの設計、画面の追加、または仕様の変更についてアイデアやメモを提示したとき。
- **動作ルール**: 
  - `gdd-creator` のプロンプトをロードして動作します。
  - いきなり仕様書を生成せず、進捗カウンター（例: `[1/3]`）を提示して最大3ターンの一問一答インタビューを行い、最終的に `docs/specs/[機能名]-SSoT.md` を出力・更新します。

### 1.2. 仕様書の品質監査 ──> `gdd-validator` (Suárez)
- **トリガー**: `*SSoT.md` が作成・更新されたとき、または仕様のレビューやバグ・考慮漏れのチェックを指示されたとき。
- **動作ルール**: 
  - `gdd-validator` のプロンプトをロードして動作します。
  - `3.3. 不変条件` に記載されている全項目に対する監査結果（PASS/FAIL）を網羅したチェックリストと、エラーID（`[ERROR-XX]`）を含むテストレポートを、隠しフォルダ配下の `.agents/gdd-reports/[機能名]-gdd-validator-report.md` に出力します。

### 1.3. 仕様の自動修復 ──> `gdd-resolver` (Neymar)
- **トリガー**: Validator が `FAIL` を出したとき、または仕様書の矛盾や指摘された不備の修正を指示されたとき。
- **動作ルール**: 
  - `gdd-resolver` のプロンプトをロードして動作します。
  - レポートに記載されているすべての `[ERROR-XX]` を論理的に解消した最新の `[機能名]-SSoT.md` を上書き保存し、再度 Validator を実行するように促します。

---

## ⚽ 自動連携（パス回し）のガイドライン

1. **ナビゲーションの厳守**:
   - 各スキルの実行完了時は、メッセージの最末尾に、ユーザーが次にどのコマンドやアクションを指示すべきかを「⚽ Next Pass (推奨)」として必ず提示してください。
2. **自律エージェントでの自動ループ**:
   - 自動実行が許可されている環境（Cline等）では、指示を受けるだけで `仕様作成 -> 監査 -> 修復` のループを自律的に回し、最終判定が `PASS` になったクリーンな仕様書を作成した段階で、ユーザーに最終検収を求めてください。
